# 你的 AI Agent 每天做研究，但它什么都没记住

昨天让 Agent 分析了个人养老金政策，聊了一个多小时，整理出七八个关键概念，还找到两篇论文里的数据矛盾。今天接着问同样的领域，又从零开始搜。

试过 RAG。把文档丢进去，每次提问检索一遍，拼出一个回答。能用，但你会发现一个问题：上周读过的十篇材料，这周再问，它还是在"重新发现"。没有什么被积累下来。知识不长在 Agent 身上。

Agent 不笨，可能比你读得快、总结得好、分析得有条理。但它是个金鱼，转一圈回来什么都不记得。

---

## 一、问题不在能力，在载体

现在的 Agent 不缺 Skill。教它拉数据、写报告、做分析、搜论文，每个都好使，做完这一单就好使。

问题是 Skill 之间没有连接，做完的成果没有落点。

上午用搜索 Skill 找了三篇养老金政策文件，下午用分析 Skill 写了一份比较报告。明天想在这个基础上继续深化，不好意思，明天是新对话，Agent 不记得上午搜了什么、下午写了什么。你得重新描述背景、重新提供材料、重新等它"理解"一遍。

Skill 教 Agent 怎么做事，但做事的产出散落在一次次对话里，没有一个地方把它们串起来。

缺的不是更多 Skill，是一个载体，让每次研究的产出都能沉淀下来，下次接着用。

---

## 二、编译，而非检索

这个载体是什么？一个 Agent 自己维护的 wiki。

Agent 读完一份材料之后，主动把里面的关键信息写进 wiki 已有的页面，和旧信息逐页比较，该更新的更新，有矛盾的标出来，全新的概念建新页面。

这个过程叫编译。和 RAG 放在一起对比：

| | RAG | 编译 |
|---|---|---|
| 什么时候处理信息 | 查询时 | 收到材料时 |
| 知识存在哪 | 散在原始文档里 | 写进结构化的 wiki 页面 |
| 同一个问题问两次 | 重新检索、重新推导 | 直接读编译好的页面 |
| 两份材料矛盾了 | 看那次检索碰上哪份 | 明确标注 contested，两种说法并列 |

编译的核心是拿到新材料后的三路比较：

| 新旧对比结果 | Agent 怎么处理 |
|---|---|
| 新信息和已有结论一致 | 强化：追加来源引用，提高置信度 |
| 新信息有更新的数据或更权威的来源 | 更新：改写页面，旧结论进历史记录 |
| 两边矛盾，分不出谁对 | 冲突：并列保留，标记 contested |

Tobi Lutke（Shopify CEO）去年提过这个想法：让 LLM 维护一个持久的 wiki，别每次查询都从文档堆里现找。Karpathy 的 autoresearch 也走了类似的路，让 Agent 自己跑研究循环。

我做的事情是把这个想法落成了一个可执行的 Skill，然后跑通了。

---

## 三、实测：从零积累一个领域

我用"个人养老金制度"做了一轮完整测试，5 个阶段，从零开始积累。

**冷启动。**

告诉 Agent："研究个人养老金制度，帮我建立知识框架。"

Agent 自己搜到了国办发〔2022〕7号和人社部实施办法两份政策文件，读完之后创建了 personal-pension wiki：7 个概念页面（EET 税制、账户唯一性、缴费上限、投资范围……）、1 个实体、2 个来源摘要。

wiki 还只是个骨架，但结构已经有了。

**深化。**

"找一些关于参与意愿的研究。"

Agent 搜到几篇学术论文，ingest 进来。然后它发现了一个矛盾：一篇调查说 77.8% 的人有参与意愿（样本来自一线城市），另一项数据显示实际缴费率只有 25%。

Agent 没有硬选一个，而是把两个数字都留着，标记为 contested，注明了各自的来源和可能的偏差原因。

碰到矛盾不调和，并列呈现，等更多证据。三路比较在实际运行中就是这个样子。

**国际比较。**

"对比一下美国的 IRA 制度。"

Agent 搜了 IRA 的材料，建了 ira-usa、auto-enrollment 等新概念页，并把它们和已有的中国制度页面用 wikilink 关联起来。

**学以致用。**

"如果我想提高参与率，政策应该怎么设计？"

Agent 读 wiki，读它自己编译过的页面。回答里引用了 `[[enrollment-friction]]`（开户流程的摩擦成本）、`[[tax-incentive-effect]]`（EET 税优对低收入群体激励有限）、`[[ira-usa]]`（美国 auto-enrollment 后参与率从 40% 涨到 94%）。

然后它说："wiki 里缺乏 35 岁以下群体行为特征的专门研究，建议补充。"

它在告诉你知识缺口在哪。

最终数据：22 个页面，8 个结构化数据点，23 条关系边，2 处 contested。全部 schema 校验通过。5 轮对话，逐步积累出来的。

## 四、数据该在数据库里

最初设计时，结构化数据（管理规模多少亿、覆盖率多少）写在 markdown 文件的 YAML 头部。能用，但很快碰到问题：想查"所有页面中哪些数据是 contested 的"，得扫全部文件；想看某个指标的时间线，得逐个页面翻 history 字段。

后来改了。markdown 只负责叙事分析，数字全进 SQLite。

```python
store.upsert_data("enrollment-friction", "实际缴费率", 25, "%", "2023-10", "2024-survey")
# 如果同字段已有旧值，自动写入 history 表，返回旧记录
```

每个数字必须有 source_slug，指向哪个来源页面。这条规则没有例外。一旦允许"来源不明"的数字进 wiki，后续查询引用它时就没有审计链，错误数据会越滚越大。

wiki 是叙事层，SQLite 是数据层。Agent 在 markdown 里写分析，在数据库里写数字。两层分开，各干各的。

## 五、逻辑纠错

Agent 从研报里提取出一个 PensionFund 实体，写进了 wiki，只有名称和规模。看起来没问题？

我在 lint 里加了一层可选的外部校验：连接 FIBO（金融行业语义本体标准，627K 条推理三元组），用 SPARQL 查询这个实体类型在标准本体里应该有哪些必要关系。

查询结果：PensionFund 有 22 条约束，必须有 management company、depository、supervisory authority、fund investment policy……

Agent 写的那个 PensionFund 页面缺了 9 条。

这件事有意思在哪：它纠的不是错别字，也不是数字错误，而是逻辑上的不完整。你说这是一个 PensionFund，但按照行业标准定义，一个 PensionFund 至少要有管理人、托管人、监管方。页面连这些关系都没建，说明知识还有缺口。

种子词表是冷启动时用的静态参考，防止 Agent 把"计划"和"基金"搞混。FIBO SPARQL 是运行时的动态校验。两层都是可选的插件，不装也能跑，装了 lint 会更深。
